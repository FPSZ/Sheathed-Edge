# 可控式训练工程路线图 V2

副标题：结合 Axolotl / Inspect AI / AgentEvals / CAISI 的离线可控训练主线

## Summary

这份文档把当前仓库中已经存在的训练数据、评测目录、技能体系和 Axolotl 脚手架，收敛成一条可执行的训练工程路线。目标不是做 agent 自学习，而是建立一套人工可控、离线可运行、可复盘、可逐步扩展的训练闭环。

固定决策如下：

- 训练策略固定为 `skills-first + eval-first + human-in-the-loop + SFT second`
- 首轮题型固定为 `reverse`
- 第一阶段不做自学习、不做自动回灌、不做线上日志直喂训练
- 全流程要求支持离线环境
- 当前认可的核心闭环固定为：学生做题 -> teacher 复做 -> 人工差异审查 -> 更新 skills / 进入训练候选 -> 回归评测

这份路线图解决 4 个核心问题：

1. 为什么当前阶段不做自学习，而做可控训练
2. 当前仓库哪些目录继续沿用，哪些是新增训练层
3. 一道题如何从原始轨迹流向 `skills`、`SFT` 和 `eval`
4. 什么时候只改 `skills`，什么时候才允许进入训练

## Why Controlled Training

当前系统暴露出来的主要问题，不是“完全不会做题”，而是“过程不稳定”：

- 工具选型不稳
- shell 适配容易错
- 路径与编码容易出问题
- 出错后的 fallback 不够快
- 最终证据整理质量不稳定

这类问题如果直接做自学习或自动回灌，会把噪声、偶然成功和错误恢复失败一并吸进训练集，最终污染模型行为。相反，当前更适合的路线是：

- 先用 `skills` 固定流程
- 再用 `eval` 看流程是否真的改善
- 再用人工 review 过的样本做小规模 `SFT`

固定原则：

- `skills` 是第一道修正手段
- `SFT` 是第二层增强，不替代人工审查
- `eval` 是放行门禁，不是训练后的附属品
- `reverse` 先打穿，再复制到 `pwn / web / awdp`

## External Stack Role Map

### 1. Axolotl

GitHub: <https://github.com/axolotl-ai-cloud/axolotl>

定位：

- 训练主线
- 负责 `SFT / LoRA / QLoRA`
- 负责读取转换后的训练数据、执行训练、导出 checkpoint

负责：

- 训练配置
- 数据加载
- dry-run
- 小样本试跑
- 正式训练配置收口

不负责：

- 在线推理
- 题目评测编排
- 自动纠错闭环
- 人工 review 替代

### 2. Inspect AI

GitHub: <https://github.com/UKGovernmentBEIS/inspect_ai>

定位：

- 评测执行框架
- 负责多轮对话、工具调用、预算控制、可复跑 eval

负责：

- reverse dev/eval 执行
- 多轮 agent 任务回放
- 预算、回合数、工具调用上限
- 结果日志与可复跑评测

不负责：

- 训练样本清洗
- 最终训练执行
- 直接代替人工 review

### 3. AgentEvals

GitHub: <https://github.com/langchain-ai/agentevals>

定位：

- 过程监督与轨迹评分组件
- 负责评价“过程是否合理”，不是只看最后做没做对

负责：

- judge 工具选择是否合理
- judge 步骤顺序是否合理
- judge fallback 是否及时
- judge 证据整理质量是否达标

不负责：

- 代替人工审查结论
- 自动决定样本能否入库

### 4. CAISI Cyber Evals

GitHub: <https://github.com/usnistgov/caisi-cyber-evals>

定位：

- 安全题 benchmark 组织参考
- 借鉴 `benchmark / task / sample / budget / agent` 的结构化组织方式

负责：

- 提供评测工程组织参考
- 提供安全任务 benchmark 思路

不作为第一阶段硬依赖：

- 不要求当前仓库直接改成它的全套格式
- 不要求第一阶段就接入完整生态

### 5. inspect_evals

GitHub: <https://github.com/UKGovernmentBEIS/inspect_evals>

定位：

- 现成 eval 目录和任务组织参考
- 用来借鉴 reverse 任务定义、评测样板和结果组织方式

第一阶段策略：

- 参考其结构
- 不强依赖其仓库
- 不把当前项目硬改成它的目录协议

## Repo Integration

- `Feed/Histroy`
  - 只放原始轨迹
- `Datasets/reverse`
  - 放清洗后的 case、review、skill-delta
- `Eval/reverse`
  - 放 dev/eval 题目、预算、结果、评分
- `Training/axolotl`
  - 放训练配置、转换脚本、训练输出
- `Core/awdp/skills`
  - 放通用技能
- `Plugins/pwn/skills`
  - 放 reverse 现阶段依附的技能层
- `Docs/Plan/训练评测`
  - 放训练总路线图、专项计划和评测计划

建议新增但不要求本轮立刻完全实现的骨架：

- `Training/inspect/`
  - 放 Inspect AI 任务定义、runner、预算模板
- `Training/reports/`
  - 放训练前后对比、失败模式聚类、覆盖度统计
- `Training/reviews/`
  - 若后续不继续沿用 `Datasets/reverse` 作为 review 真源，可在此集中放审查规范与审查模板

## Stable Artifacts

当前训练工程固定围绕 4 类主工件展开，不再改名：

- `ReverseCaseRecord`
- `TraceReviewRecord`
- `SkillDeltaRecord`
- `ReverseEvalRecord`

### Training Input

`Training/axolotl/datasets/*.jsonl`

每条训练样本至少带：

- `case_id`
- `source_split`
- `review_status`
- `skill_context`
- `error_tags`
- `teacher_verified`

训练样本默认采用：

- 多轮对话
- 工具调用记录
- 结构化过程标签

至少保留的过程字段：

- `phase`
- `goal`
- `chosen_tool`
- `why_this_tool`
- `expected_evidence`
- `observed_result`
- `fallback_if_fail`

### Eval Output

`Eval/reverse/...`

至少记录：

- `solve_rate`
- `time_to_first_useful_action`
- `invalid_tool_call_rate`
- `shell_mismatch_rate`
- `avg_tool_calls`
- `error_recovery_turns`
- `truncated_output_incidents`
- `final_evidence_quality`

### Admission Rule

任何 case 只有同时满足以下条件，才能进入 `sft-ready`：

- `review_status=approved`
- `teacher_verified=true`
- 不属于 `eval` 污染样本
- 没有看过答案后再伪装成 held-out eval

## Operator Workflow

### 1. 原始轨迹进入系统

来源固定为：

- 本地 Open WebUI / Gateway / Tool Router 的真实做题轨迹
- 手工导出的历史会话

原始轨迹只放 `Feed/Histroy`，不直接进训练。

### 2. 学生模型做题

固定要求：

- 独立完成
- 保留完整工具轨迹
- 保留失败恢复过程
- 保留最终结论与证据

### 3. Teacher 复做同题

固定要求：

- teacher 不参考 student 过程
- 重新独立求解
- 输出更高质量的“理想过程”

### 4. 人工差异审查

人工审查至少回答：

- student 错在哪
- 是知识问题还是流程问题
- 哪一步该更早切换工具
- 哪一步该更快 fallback
- 哪些错误应该沉淀成 skill
- 哪些样本值得进入 SFT 候选

### 5. 产出三类工件

- `review`
- `skill delta`
- `sft-ready sample`

注意：

- `sft-ready` 不是自动生成
- 必须经过人工 review 放行

### 6. 先改 skills 再跑 dev

每次 case review 后，先判断能否通过 `skills` 修复。

优先修这类高频问题：

- `shell_mismatch`
- `path_or_encoding_failure`
- `missing_dependency_fallback`
- `powershell_syntax_error`
- `low_signal_exploration`
- `bad_error_recovery`
- `wrong_tool_priority`
- `poor_evidence_summary`

### 7. 只有在 eval 不回归时才进入训练

固定门禁：

- 先看 dev 是否改善
- 再看 eval 是否稳住
- 只涨 train、不涨 dev/eval，视为无效优化

### 8. 再做小规模 SFT

当一批 case 已经通过：

- skills 修订
- review 审批
- dev 回归
- eval 无污染

才允许转换成 Axolotl 数据，做小规模 `LoRA / QLoRA` 试跑。

## Phase Roadmap

### Phase 0：基线定型

目标：

- 收口目录
- 收口字段
- 收口 split 规则
- 收口 review 模板

验收：

- 1 个 reverse case 能完整跑通
- 四类工件能稳定落盘
- train/dev/eval 不串

### Phase 1：评测先行

目标：

- 用 `Inspect AI` 跑 reverse dev/eval
- 用 `AgentEvals` 补过程评分
- 先建立稳定评测，不急着训练

验收：

- 同一批题可重复回放
- 工具预算、回合预算、过程评分可稳定输出
- 能明确比较 skills 更新前后差异

### Phase 2：skills-first

目标：

- 针对高频错误更新技能
- 重点修流程稳定性

第一批技能建议继续固定为：

- `reverse-intake-and-path-normalization`
- `reverse-shell-adapter`
- `reverse-archive-and-binary-triage`
- `reverse-static-analysis-escalation`
- `reverse-hypothesis-loop`
- `reverse-final-report`

验收：

- `shell_mismatch_rate` 下降
- 路径/编码失败下降
- fallback 更快
- 平均工具调用更少但更有效

### Phase 3：小规模 SFT

目标：

- 把通过 review 的样本转成 Axolotl 数据
- 做小规模 `LoRA / QLoRA`
- 验证训练是否真的提升过程稳定性

验收：

- `Axolotl` dry-run 成功
- 小样本训练能产出 checkpoint
- 训练后 dev 改善且 eval 不回归

### Phase 4：安全题扩展

目标：

- 若 reverse 跑通，再复制到 `pwn / web / awdp`
- 更系统地引入 `CAISI` 风格 benchmark 设计

验收：

- 新题型沿用同一套 review / skill / eval / training 工程协议
- 不再为每个题型重写整套流程

## Now / Later / Deferred

### Now

当前优先事项：

- 把这条可控训练主线写清楚
- 用现有 reverse 样本打穿 1 个完整闭环
- 固定评测门禁
- 固定训练入库门禁

### Later

后续逐步推进：

- `Training/inspect/` 任务定义与 runner
- reverse pilot 数据集扩充
- 小规模 Axolotl 试跑
- 训练前后对比报告

### Deferred

明确延期，不阻塞当前路线：

- 自学习 / 自动进化
- 自动从线上日志吸样
- DPO / GRPO / RL
- 多题型同时开训
- 视觉能力训练

## Prohibitions

固定禁止事项：

- 禁止把原始自由对话直接当训练集
- 禁止把已看过答案的 eval 题回灌训练
- 禁止把线上生产日志自动并入训练集
- 禁止跳过人工 review 直接生成 `sft-ready`
- 禁止让 eval 框架兼做训练框架
- 禁止把过程 judge 结果直接等同于人工审查结论

## Acceptance Gates

只有以下条件全部满足，才允许从“skills-first”进入“小规模 SFT”：

- 至少 1 个 reverse case 全流程打通
- `Inspect AI` dev/eval 可稳定复跑
- `AgentEvals` 过程评分已接入
- 至少积累 30 个经人工校正的 reverse case
- 主要失败模式已开始收敛，而不是每题都换一种新错法
- 训练样本、评测样本和技能规则三者已明确隔离

## Test Plan

### 数据链路

- 1 个现有 reverse case 能从原始轨迹整理成完整四类工件
- 重复 `case_id`、非法 split、缺关键字段时会被校验脚本拦住
- train/dev/eval 不串

### 评测链路

- 同一批 reverse 题可以用 `Inspect AI` 稳定复跑
- 过程评分可由 `AgentEvals` 输出，不只剩最终是否做对
- skills 更新前后可得到可比对结果

### 训练链路

- `Axolotl` 能完成 dry-run
- 小样本 SFT 能启动并产出 checkpoint
- 离线环境下不依赖外部在线服务

### 控制链路

- 未 review 的 case 不进入训练
- 看过答案的题不能进 `eval`
- 线上日志不会被自动回灌
- judge 结果不会绕过人工审批

### 路线图验收

- 另一个工程师读完文档后，无需补关键决策就能开始搭脚手架
- 外部项目职责边界清楚，不会出现 `eval` 和 `training` 混用
- 文档内容能直接映射到当前仓库目录，而不是空中方案
