# AI操作指南

## 1. 定位与目标

这份指南服务于本地离线 AWDP 系统，不是通用 AI 使用说明。

本项目里，AI 的定位是：

- 按既定架构工作，不自由发挥
- 默认运行在 `awdp` 主模式下
- 按题目需要叠加 `web` / `pwn` 子插件
- 为架构设计、后端实现、工具接入、题目分析提供可执行输出

这份文档用于统一以下行为：

- 怎么提问
- 怎么拆任务
- 怎么调用工具
- 怎么记录结果
- 怎么复盘失败

## 2. AI 常见错误清单

### 2.1 指令过于模糊

错误模式：

- 目标不明确
- 成功标准缺失
- 输出格式未指定

在本项目中的典型表现：

- 让 AI “设计一下架构” 但没有约束技术栈、模式系统和工具边界
- 让 AI “看看怎么做” 结果输出泛泛方案

防范规则：

- 每次提问必须写清：
  - 目标
  - 约束
  - 输出格式
  - 成功标准

### 2.2 指令过多且互相冲突

错误模式：

- 一次 prompt 同时要求设计、实现、评审、优化
- 约束之间互相打架

在本项目中的典型表现：

- 同时要求“最简单”和“最极致性能”
- 同时要求“单模型”又要求“多模型协同”

防范规则：

- 一次只解决一个核心问题
- 决策问题和实现问题分开问

### 2.3 上下文不足

错误模式：

- 没给现有架构
- 没给目录结构
- 没给硬约束

在本项目中的典型表现：

- 不告诉 AI 当前是 `Go Gateway + Rust Tool Router`
- 不告诉 AI 已经固定 `awdp` 主模式

防范规则：

- 先给系统背景、技术栈、路径、端口、模式系统
- 不要求 AI 猜现状

### 2.4 把所有任务都扔给单次大 prompt

错误模式：

- 想一次性得到完整系统
- 不分阶段验证

在本项目中的典型表现：

- 一次 prompt 同时要求网络、状态机、schema、接口、部署、测试

防范规则：

- 复杂任务拆阶段
- 先定决策，再定接口，再定实现

### 2.5 工具权限过大

错误模式：

- 工具没有白名单
- 路径无限制
- terminal 可执行任意 shell

在本项目中的典型表现：

- 让模型直接拼接 shell 命令
- browser 不设目标约束

防范规则：

- 所有工具默认拒绝
- 仅按主模式和插件白名单放行
- terminal 只走模板，不走自由 shell

### 2.6 工具边界不清

错误模式：

- 两层系统做同一件事
- 内置逻辑和工具调用重复

在本项目中的典型表现：

- Gateway 检索和 retrieval tool 职责混淆

防范规则：

- Gateway 负责预检索和上下文拼装
- retrieval tool 只用于模型显式补充检索

### 2.7 复杂编排过早上量

错误模式：

- 还没做单链路验证就上多代理、多阶段、多工具

在本项目中的典型表现：

- WebUI 还没接通就开始讨论复杂 agent flow

防范规则：

- 先小闭环，再扩大复杂度
- Phase 1 先做最小流式可用链路

### 2.8 忽视资源约束和上下文膨胀

错误模式：

- 不考虑单模型上下文上限
- 不考虑工具调用耗时
- 不考虑统一内存占用

在本项目中的典型表现：

- 默认把上下文开到很高
- 并发参数写死过高

防范规则：

- 默认以稳定和延迟为先
- 高并发和超长上下文只在压测后开启

### 2.9 未做失败回退和审计

错误模式：

- tool_call 校验失败后没有回退策略
- 工具失败没有记录

在本项目中的典型表现：

- schema 校验失败直接丢失上下文
- 工具超时后不写日志

防范规则：

- 所有失败都要有结构化错误
- schema 失败只允许一次 repair retry
- 再失败降级为纯 answer

### 2.10 做完不记录

错误模式：

- 做了决策不写日志
- 改了计划不更新待办

在本项目中的典型表现：

- 架构已修订，但没人知道修了什么
- 下一次会话需要重新推断上下文

防范规则：

- 做完必须写日志
- 做完必须更新待办
- 没有日志和待办更新，视为任务未完成

## 3. 本项目操作原则

- 先读计划，再动实现
- 先确认模式，再调用工具
- 先小闭环，再扩大复杂度
- 一次只解决一个核心问题
- 先保留证据，再下结论
- 工具失败必须显式记录
- 未记录不得切任务
- 未更新待办不得视为完成
- 所有输出优先结构化
- 所有实现先满足可回退，再追求更复杂能力

## 4. 标准工作流

### 4.1 架构设计任务

流程：

1. 读现有计划文档
2. 确认当前未定义问题
3. 单点提问，只讨论一个决策域
4. 把结论写回架构文档
5. 更新待办
6. 写日志

做完必须记录：

- 解决了什么未定义问题
- 文档改了哪些结论
- 下一步实现依赖是什么

### 4.2 后端实现任务

流程：

1. 确认目标接口或服务边界
2. 先实现最小骨架
3. 做单链路验证
4. 通过后再加工具或插件逻辑
5. 更新待办
6. 写日志

做完必须记录：

- 新增了什么服务或接口
- 当前能跑到哪一步
- 哪些依赖还没满足

### 4.3 MCP / 工具接入任务

流程：

1. 明确工具 scope
2. 明确 schema
3. 明确路径 / URL / 超时 / 重试边界
4. 接入最小案例
5. 验证失败回退
6. 更新待办
7. 写日志

做完必须记录：

- 工具名
- scope
- 安全限制
- 已验证场景

### 4.4 漏洞分析 / 题目分析任务

流程：

1. 默认加载 `awdp`
2. 按题目需要叠加 `web` / `pwn`
3. 先分析攻击面
4. 再决定是否调用工具
5. 保留证据与结论
6. 记录失败 case

做完必须记录：

- 模式组合
- 使用了哪些工具
- 得到了什么证据
- 是否形成 patch / exp / 归因

### 4.5 文档整理任务

流程：

1. 先确认文档定位
2. 避免和现有文档重复
3. 写成规则或结构，不写空话
4. 更新待办
5. 写日志

做完必须记录：

- 新增或修改了哪份文档
- 解决了哪个信息缺口
- 后续谁要消费这份文档

## 5. 日志记录规范

当前日志文件：

- [2026-03-27.txt](D:\AI\Local\Docs\Log\2026-03-27.txt)

固定模板：

```text
[时间]
任务：
结果：
修改：
发现的问题：
下一步：
状态：完成 / 进行中 / 阻塞
```

记录规则：

- 一次工作会话可记录多块
- 新块追加到文件末尾
- 不覆盖旧记录
- 只做调研没改文件，也必须记
- 阻塞项必须写清阻塞原因和依赖

以下动作结束后必须写日志：

- 写完或改完一个计划文档
- 完成一个后端阶段骨架
- 接通一个服务链路
- 确认一个技术路线
- 发现一个会影响架构上限的问题

日志风格要求：

- 短句
- 事实
- 可回看
- 不抒情

## 6. 提问与提示词方法论

固定规则：

- 大任务拆小
- 上下文先给硬约束
- 问题必须单一目标
- 输出必须明确格式
- 需要 schema / struct / 伪代码时必须写明
- 先问决策，再问实现

### 6.1 坏提示词 vs 好提示词

#### 架构设计

坏提示词：

- 帮我设计整个 AI 系统

好提示词：

- 在现有 `Go Gateway + Rust Tool Router` 架构下，只补全 `tool_call` 状态机，输出状态表、Go 结构体和主循环伪代码

#### Tool Router

坏提示词：

- 帮我写工具路由

好提示词：

- 为 `POST /internal/tools/execute` 设计请求/响应 JSON、Rust 结构体、错误码和超时策略，必须兼容 `awdp/web/pwn` scope

#### mode / plugin schema

坏提示词：

- 帮我设计配置文件

好提示词：

- 设计 `mode.json` 和 `plugin.json` 的 JSON schema，必须覆盖 prompt 路径、skills 路径、tool scopes、retrieval roots、eval tags 和 merge 规则

#### 漏洞分析

坏提示词：

- 看看这个题怎么做

好提示词：

- 在 `awdp+web` 模式下，先给攻击面排序，再列证据，再决定是否需要调用 browser 或 retrieval 工具

#### patch 生成

坏提示词：

- 帮我修这个漏洞

好提示词：

- 基于现有代码和日志，给出最小 patch、回归风险、验证步骤，并标记仍需补充的证据

## 7. 工具与模式使用纪律

- 默认模式永远是 `awdp`
- 只有题目需要时才叠加 `web` / `pwn`
- 不允许一开始就全插件全工具
- Gateway 内置检索用于上下文拼装
- retrieval tool 只用于模型主动补检索
- terminal 只能走模板，不允许自由 shell
- browser 只能访问静态策略与动态目标集共同允许的目标
- 工具失败后必须返回结构化错误，不能静默忽略

---

## 8. 复盘与迭代纪律

每次阶段结束必须做：

- 补日志
- 更新待办复选框
- 标记阻塞
- 记录失败 case

固定规则：

- 没有更新待办和日志 = 任务未完成
- 失败 case 必须进入后续评测或文档回灌
- 发现架构上限风险时，优先修文档和待办，再开实现
