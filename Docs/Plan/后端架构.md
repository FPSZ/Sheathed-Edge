# 后端优先实施计划：Go Gateway + Rust Tool Router

## Summary

后端先行，先不碰前端深度定制。V1 后端按混合语言硬分层实施：

- `Go`：`Agent Gateway`
- `Rust`：`Tool Router`
- `Python`：仅做 MCP / 安全工具适配层，不进入核心控制面
- 运行环境：`WSL2 Ubuntu`
- 推理后端：`llama-server` 运行在 Windows，本机由 WSL2 访问

这套方案的目标不是先跑起来，而是给后续系统上限留足空间：高性能推理、硬安全边界、清晰扩展面、可替换工具层、可长期演进。

---

## 1. 固定后端分层与职责

### 1.1 Agent Gateway（Go）

职责：

- 对外提供唯一 OpenAI-compatible 接口，供 Open WebUI 访问
- 注入 `awdp` 主模式
- 按需叠加 `web / pwn` 插件
- 做用户请求发出前的检索拼装
- 构造对 `llama-server` 的推理请求
- 解析 `JSON action envelope`
- 校验 `tool_call` 是否符合 schema 和当前 scope
- 调用 `Tool Router`
- 收集工具结果并发起二次推理
- 记录会话日志、检索日志、模式日志、失败归因

不负责：

- 不直接执行工具
- 不直接起 MCP 进程
- 不做工具权限治理

### 1.2 Tool Router（Rust）

职责：

- 作为唯一工具治理边界
- 加载 `D:\AI\Local\MCP\registry.json`
- 校验工具名、scope、参数 schema、路径限制、URL 限制、超时、重试策略
- 启动 / 连接 MCP 进程
- 管理 `stdio` / HTTP MCP 通道
- 结果截断、摘要、错误归一
- 记录工具执行与审计日志

不负责：

- 不做模式推理
- 不做 prompt 拼装
- 不直接接 UI
- 不直接访问 `llama-server`

### 1.3 Python MCP Adapters

职责：

- 实现具体 MCP server 和安全工具桥接
- V1 包括：
  - `filesystem/search`
  - `retrieval`
  - `browser`
  - `radare2`
  - `terminal:web`
  - `terminal:pwn`

约束：

- Python 只能被 Tool Router 调用
- Python 不进入核心控制面

### 1.4 llama-server（Windows）

职责：

- 加载唯一模型 `DeepSeek-R1-Distill-Llama-70B-Q4_K_M.gguf`
- 提供 OpenAI-compatible 推理接口
- 只负责推理，不做业务逻辑

## 2. 推理服务部署基线

当前最大风险不是模型不够强，而是硬件没有被充分利用。V1 文档中固定 `llama-server` 启动基线，避免默认参数限制性能。

### 2.1 启动参数策略

默认策略：

- 尽量全层 GPU offload
- 默认上下文：`32768`
- 打开 `flash-attn`
- 先保证单会话稳定和低延迟
- 并发参数通过实测调优，不在 V1 写死为高值

### 2.2 推荐启动参数

Windows 上 `llama-server` 启动参数建议：

```powershell
llama-server `
  --model D:\AI\Local\Models\Deepseek\DeepSeek-R1-Distill-Llama-70B-Q4_K_M.gguf `
  --host 127.0.0.1 `
  --port 8080 `
  --n-gpu-layers 999 `
  --ctx-size 32768 `
  --flash-attn
```

说明：

- `--n-gpu-layers 999` 表示“尽量全量 offload”，不要把 `99` 当成固定魔法数字
- `--ctx-size 32768` 作为 V1 默认
- `65536` 只作为实验档，不作为默认
- `--parallel` 不在文档里写死，后续压测后再定

### 2.3 Gateway 和部署层的边界

- 启动参数属于部署层，不属于 Gateway client 配置
- Gateway 只知道：
  - `base_url`
  - 健康检查
  - 超时
- Gateway 不负责决定 `n-gpu-layers`、`ctx-size`、`flash-attn`

## 4. 固定数据流

### 4.1 主数据流

- 用户 -> Open WebUI -> Gateway
- Gateway -> 检索拼装 -> `llama-server`
- 模型输出 `tool_call` -> Gateway 校验 -> Tool Router
- Tool Router -> MCP Adapter
- 结果返回 Gateway -> 二次推理 -> 最终答复

### 4.2 Open WebUI 接入方式

- Open WebUI 只接 `Agent Gateway`
- 只暴露一个逻辑模型别名：`awdp-r1-70b`
- WebUI 不直接感知：
  - GGUF 文件路径
  - MCP 列表
  - 插件组合细节

## 5. 模型动作协议与 fail-closed 规则

动作协议固定使用：

- [action-envelope.schema.json](D:\AI\Local\Agent\action-envelope.schema.json)

只允许两类动作：

- `answer`
- `tool_call`

### 5.1 基本校验规则

- `type=answer`：不触发工具调用
- `type=tool_call`：必须校验
  - 工具名是否在白名单中
  - 参数是否满足 schema
  - 路径 / URL 是否在允许范围

### 5.2 fail-closed 策略

当模型输出不符合 schema 时：

1. 第一次校验失败
2. Gateway 注入固定错误提示，做 `1` 次 repair retry
3. 若 repair 后仍失败：
   - 不进入工具调用分支
   - 降级为纯 `answer`
   - 记录审计日志

当 `tool_call` 参数校验失败时：

1. 返回结构化校验错误给 Gateway
2. Gateway 最多做 `1` 次 repair retry
3. 再失败则降级为纯 `answer`

不允许：

- 无限重试
- 模型输出非法 `tool_call` 后仍直接执行工具

## 6. 模式与插件系统

### 6.1 主模式

主模式固定为 `awdp`，永远加载：

- 系统提示词
- 输出格式
- 风险偏好
- patch 优先级
- 回归检查模板
- 默认工具白名单

### 6.2 子插件

V1 固定：

- `web`
- `pwn`

### 6.3 允许组合

- `awdp`
- `awdp+web`
- `awdp+pwn`
- `awdp+web+pwn`

### 6.4 优先级规则

- `awdp` 永远最高
- 子插件只能补充，不能覆盖主模式
- 复合题允许同时启用 `web` 与 `pwn`

### 6.5 加载来源

Gateway 从以下配置读取：

- [mode.json](D:\AI\Local\Core\awdp\mode.json)
- [plugin.json](D:\AI\Local\Plugins\web\plugin.json)
- [plugin.json](D:\AI\Local\Plugins\pwn\plugin.json)

加载内容：

- prompt 片段
- skills 片段
- tool scopes
- retrieval roots
- eval tags

## 7. 检索职责拆分

`retrieval` 现在在系统里有两种角色，必须明确拆开。

### 7.1 Gateway 内置检索

用途：

- 用户请求发出前的上下文拼装

实现：

- `SQLite FTS5`
- `ripgrep` 兜底

特点：

- 系统默认行为
- 不需要模型主动发起
- 只由 Gateway 调用

### 7.2 retrieval MCP Tool

用途：

- 模型在 `tool_call` 阶段主动要求进一步检索

特点：

- 显式工具调用
- 经过 Tool Router 白名单和参数校验
- 与 Gateway 内置检索不是同一职责

### 7.3 设计结论

两者并存是合理的，但必须区分：

- Gateway 检索：预检索、上下文拼装
- retrieval tool：模型主动补充检索

## 8. MCP 接入与 Tool Router 设计

### 8.1 注册来源

所有 MCP server 定义集中放在：

- [registry.json](D:\AI\Local\MCP\registry.json)

Tool Router 启动时读取注册表，不从 UI 动态发现。

### 8.2 最低字段要求

每个工具至少定义：

- `name`
- `type`
- `command` 或 `url`
- `workdir`
- `timeout_ms`
- `retry`
- `allowed_paths`
- `allowed_hosts`
- `plugin_scope`
- `parameter_schema`
- `enabled`

### 8.3 默认工具路由

`awdp` 默认工具：

- `filesystem/search`
- `terminal`
- `retrieval`

`web` 插件追加：

- `browser`
- `terminal:web`

`pwn` 插件追加：

- `radare2`
- `terminal:pwn`

### 8.4 审计字段

Tool Router 必须记录：

- `session_id`
- `mode_combo`
- `tool_name`
- `arguments`
- `duration_ms`
- `result_summary`
- `error_code`
- `error_message`

## 9. terminal 工具安全模型

当前 terminal 工具是高风险点，必须固定为模板执行模型。

### 9.1 基本原则

- 不允许任意 shell 透传
- 不允许拼接整条命令字符串
- 不允许 `bash -lc`、`sh -c`

### 9.2 正确执行方式

Tool Router 侧必须使用参数化执行：

- Rust：`Command::new(...).args([...])`

绝不允许：

- 字符串插值拼命令
- 通过 shell 包裹执行

### 9.3 允许的命令模板

`terminal:web`

- `curl`
- 本地目录检查
- 有限脚本 runner

`terminal:pwn`

- `checksec`
- `file`
- `pwntools runner`

### 9.4 模板约束

每个模板必须定义：

- 命令名
- 参数 schema
- 可选参数枚举
- 允许访问路径
- 最大输出长度

## 10. browser 工具的动态目标集

“只能访问白名单本地地址”这个说法不够，AWDP web 题通常需要访问题目环境中的动态主机和端口。

### 10.1 静态策略

registry 中定义：

- `allowed_hosts`
- `allowed_cidrs`
- `allowed_schemes`
- `allowed_ports`

### 10.2 动态目标集

Gateway 在当前会话加载 `web` 插件时，可以注入 session 级目标集：

- 题目靶机 IP
- 题目端口
- 当前允许的域名 / 主机

### 10.3 最终判定规则

Tool Router 只有在同时满足以下条件时才允许浏览器访问：

1. 工具在 `web` scope 白名单内
2. 请求目标满足 registry 静态策略
3. 请求目标属于当前 session 动态目标集

## 14. V1 实施顺序

### Phase 0：基础环境

- 安装 `llama.cpp / llama-server`
- 安装 `Open WebUI`
- 安装 `WSL2`
- 安装 `go`、`rustup + stable`、`uv`、`python3`、`sqlite3`、`ripgrep`
- 下载主模型 `DeepSeek-R1-Distill-Llama-70B-Q4_K_M.gguf`

### Phase 1：Gateway 最小服务 + WebUI 烟测

- 启动 `llama-server`
- 用最小请求验证 OpenAI-compatible 接口
- 启动 `Agent Gateway`
- 实现：
  - `GET /healthz`
  - `GET /v1/models`
  - `POST /v1/chat/completions`
- 让 Open WebUI 先接 Gateway
- 立即做 WebUI smoke test：
  - model 列表
  - SSE 流式输出
  - `model` 字段兼容性

### Phase 2：Tool Router 最小服务

- 启动 `Tool Router`
- 先接：
  - `filesystem/search`
  - `retrieval`
- 完成：
  - registry 加载
  - `resolve / execute`
  - scope 判定
  - 参数 schema 校验
  - 超时
  - 路径限制
  - 工具日志

### Phase 3：Gateway 与 Tool Router 联通

- Gateway 解析 `tool_call`
- 调用 Tool Router
- 工具结果回流后二次推理
- 验证 fail-closed 和 repair retry

### Phase 4：插件与高风险工具

- 接入 `web` 插件
- 接入 `pwn` 插件
- 接入：
  - `browser`
  - `radare2`
  - `terminal:web`
  - `terminal:pwn`

### Phase 5：日志与评测

- 打通会话日志
- 打通工具日志
- 打通评测日志
- 建首批回归 case

## 15. 验收场景

### 15.1 启动验证

- `llama-server` 单独可用
- `Agent Gateway` 可作为 OpenAI-compatible provider 被 Open WebUI 接入
- `Tool Router` 能正常注册并发现本地 MCP 服务

### 15.2 模式验证

- 纯 `awdp` 会话只加载核心 prompt 和核心工具
- `awdp+web` 会话只多出 web 插件内容和 web 工具
- `awdp+pwn` 会话只多出 pwn 插件内容和 pwn 工具
- `awdp+web+pwn` 会话可同时调用两类工具，且主规则仍来自 `awdp`

### 15.3 工具验证

- 文件检索工具能读取知识库
- `radare2` 工具能分析样例二进制
- `browser` 工具能访问当前 session 允许的目标
- 非白名单工具调用会被阻断并记录
- terminal 工具不会发生 shell 拼接执行

### 15.4 回归验证

- Open WebUI 只配置一个逻辑模型入口
- Gateway 不因插件叠加破坏核心 prompt
- 工具失败时系统仍能给出结构化答复
- schema 校验失败最多修复重试 `1` 次
- 日志中能完整还原一次会话的模式、工具、结果

## 16. 默认实现假设

- 推理后端固定为 `llama.cpp / llama-server`，不是 `vLLM`
- 主模型固定为 `DeepSeek-R1-Distill-Llama-70B-Q4_K_M.gguf`
- Open WebUI 是唯一主界面
- Open WebUI 只连接 `Agent Gateway`
- MCP 不直接散接到 Open WebUI，而是统一通过 `Tool Router`
- 本机架构优先，云训和外部训练只做附录
- V1 不引入第二个常驻模型，也不引入本地向量库
