# 后端代码改进方案

> 基于对 `gateway-go` 和 `tool-router-rs` 全量源码的审查，2026-03-29

---

## 总体评价

代码整体质量较高：分层清晰，零外部 Go 依赖，Rust 侧类型建模完整，错误处理遵循 fail-closed 原则。但存在 **3 个功能性缺陷**、**5 个设计问题**、**若干代码质量问题**，其中功能性缺陷会直接影响实际使用体验。

---

## 一、功能性缺陷（需优先修复）

> 注：流式路径当前不含工具调用和检索，属于阶段性状态（工具流尚未接入），待 Phase 3 工具闭环完成后统一处理。

### F1：`action_envelope_schema` 加载了但从未校验

**位置：** `config/config.go:44`，`envelope/envelope.go:20-33`

配置项 `action_envelope_schema` 路径被读取并规范化，但 `envelope.Parse` 只做 JSON 反序列化，不用这个 schema 做校验。这意味着模型输出了不符合 schema 的 envelope（比如缺少 `reason` 字段），也会被接受或静默忽略。

**修复方向：** 在 Gateway 启动时用 `jsonschema` 加载 schema，在 `envelope.Parse` 之后补充一次校验，校验失败走 `repairEnvelope` 路径。

---

## 二、设计问题

### D1：Gateway 日志与 Tool Router 日志的 session_id 不一致

**位置：** `orchestrator/orchestrator.go:156, 177` + `logging/session.go:51`

Gateway 自己用 `requestID`（每次请求生成的 UUID）做会话日志 key。但调用 Tool Router 时传的是 `providerResp.ID`（llama-server 返回的 chatcmpl-xxx），两者不匹配。

**后果：** 想通过 session_id 把 Gateway 日志和 Tool Router 日志关联起来分析一次完整请求，做不到。

**修复：** 把 Gateway 的 `requestID` 作为 `session_id` 传给 Tool Router，不用 `providerResp.ID`。

---

### D2：Mode 配置每次请求都从磁盘重新读取

**位置：** `mode/loader.go:48`

`Loader.Load(plugins)` 在每次 `RunTurn` 调用时都打开并读取 `mode.json`、所有 prompt 文件、所有插件配置文件。对于本地单用户场景，文件通常不会运行时变更。

**修复方向：** 在 `Loader` 初始化时预加载所有允许的模式组合并缓存，或者加文件 mtime 检查，只在文件变更时重新加载。

---

### D3：Tool Router 顶层 `allowed_paths` 配置字段未使用

**位置：** `tool-router.config.json` + `models.rs:19`

`Config.allowed_paths` 被反序列化进 `AppState`，但 `check_tool_access` 和所有 executor 只检查 `ToolEntry.allowed_paths`（来自 registry），顶层配置的 allowed_paths 完全没有参与任何决策。

**修复方向：** 要么在 `check_tool_access` 中增加对顶层 allowed_paths 的交叉校验（更严格的防御），要么删掉这个字段避免误导。

---

### D4：SQLiteIndex 配置项加载但检索层从不使用

**位置：** `config/config.go:38`, `retrieval/service.go`

`Retrieval.SQLiteIndex` 被 config 加载，但 `retrieval.Service` 完全基于 ripgrep 实现，没有 SQLite 查询路径。

当前只有 ripgrep 兜底，没有 FTS5 主路径——这与设计文档的描述不一致。

**修复方向：** 短期可在配置注释里标明"V1 仅 ripgrep，SQLite 待接入"，避免产生误解；长期按计划补上 SQLite FTS5 查询路径。

---

### D5：上游模型 ID 永久缓存，无失效机制

**位置：** `provider/client.go:189-226`

`resolveUpstreamModel` 第一次成功后把模型 ID 写入 `p.upstreamModelID`，之后永远用缓存值。如果 llama-server 重启并加载了不同的模型（如切换 profile），Gateway 不会感知。

**修复：** 给缓存加 TTL（例如 5 分钟），或者在 `Health` 检查失败后清空缓存。

---

## 三、代码质量问题

### Q1：`classifyTurn` 关键词列表过于宽泛

**位置：** `orchestrator/orchestrator.go:389-411`

词表包含 "web"、"code"、"server" 等过于宽泛的词。用户输入"推荐一个 web 框架"会触发任务模式并启动检索，即使这是个纯对话问题。

**建议：** 关键词列表应收紧为强信号词（AWDP、CTF、pwn、漏洞、payload 等），去掉 "web"、"server"、"code"、"help me" 这类无法区分对话和任务的词。

---

### Q2：`retrieval.Fragment.Text` 包含冗余的 ripgrep 原始行

**位置：** `retrieval/service.go:70-79`

`Fragment.Source` 已经是 `file:line`，`Fragment.Text` 却是完整的 `file:line:content` 原始行。注入 prompt 时：

```text
- {source}
{text}  ← 包含 file:line: 前缀，与 source 重复
```

**建议：** Text 应只存 content 部分（`parts[2]`），而不是整行。

---

### Q3：`execute_tool` 重复执行 `check_tool_access`

**位置：** `api.rs:55-56`

Gateway 的标准调用链是 `resolve_tool → execute_tool`。`resolve_tool` 已经做过 scope 校验和 schema 校验，`execute_tool` 又做了一次相同的检查。虽然有防御价值，但逻辑重复不明确。

**建议：** 保留双重校验（合理的 defense-in-depth），但在代码注释中说明"此处为二次防御"，而不是让读代码的人疑惑为什么重复。

---

### Q4：`rg --max-count` 是 per-file 限制，被当成全局限制使用

**位置：** `retrieval/service.go:44`, `executor/search.rs:52-55`

ripgrep 的 `--max-count N` 含义是"每个文件最多 N 条匹配"，不是全局 N 条。如果检索根目录有 10 个文件，可能返回最多 `N × 10` 条原始结果，然后再靠 `.take(limit)` 截断。

**建议：** 改用 `--max-total-count`（rg ≥ v13）或者删掉 `--max-count`，只靠后处理的 `.take(limit)` 全局截断。

---

### Q5：`logging.rs` 手写格里高利历计算

**位置：** `tool-router-rs/src/logging.rs:44-61`

实现了一套从 Unix 时间戳到 `YYYY-MM-DD` 的日历转换算法，而不是依赖标准库或 `chrono`。

**建议：** Cargo.toml 加 `chrono = { version = "0.4", default-features = false, features = ["clock"] }` 替换，减少自制代码的维护风险。

---

### Q6：`plugin_scope_defaults` 在 Tool Router 配置中定义但从未使用

**位置：** `tool-router.config.json`，`models.rs:20-22`

`plugin_scope_defaults` 字段被反序列化存入 Config，但没有任何代码读取它。工具 scope 判断完全靠 registry 中每个 ToolEntry 的 `plugin_scope` 字段。

**建议：** 删掉这个配置字段，或者在代码中实际使用它（作为 registry 中 scope 缺失时的默认值）。

---

## 四、缺失保护（次要）

### M1：无请求上下文长度保护

Gateway 把全部历史消息 + 系统提示 + 检索片段直接发给 llama-server，没有 token 估算和截断。长会话可能超出模型上下文窗口导致推理质量下降或报错。

**建议：** 按字符数粗估 token（1 token ≈ 3-4 字符），超过阈值时截断最早的历史消息，保留系统提示和最近 N 条。

---

### M2：`SessionEntry` 日志记录了完整消息历史

**位置：** `logging/session.go:27`

`SessionEntry.Messages` 包含 `req.Messages` 全量，这意味着日志里会有完整的组装好的系统提示（含检索注入内容），每条日志条目可能很大。

**建议：** 只记录最后一条用户消息和消息条数，而不是全量历史。

## 六、建议实施顺序

**立即处理（不改会影响日常使用）：**

1. F3：补 envelope schema 校验
2. D1：统一 session_id 传递

**近期处理（提升可靠性）：**
4. F3：补 envelope schema 校验
5. D5：模型 ID 缓存加 TTL
6. Q1：收紧 classifyTurn 关键词
7. Q2：修 Fragment.Text 冗余
8. D3：清理或使用 allowed_paths

**长期处理：**
9. D2：Mode 配置缓存
10. D4：SQLite FTS5 检索接入
11. M1：上下文长度保护
12. Q4-Q6：小代码质量清理
