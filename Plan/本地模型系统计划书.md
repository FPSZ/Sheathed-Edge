# 本地模型系统计划书

## 1. 目标与约束

这套系统的目标不是通用聊天，而是做一套围绕 `AWDP` 训练和使用的本地离线作战系统，覆盖：

- AWDP 线下赛
- `web` 攻击面分析、利用、修补
- `pwn` 二进制分析、利用链推理、exp 生成
- 复合题目下的多模式切换
- 日志、源码、补丁、脚本、服务排障

当前硬约束：

- 本地只能长期保留 **一个** 主模型
- 本地主模型固定为 `70B Q4`
- 不引入常驻辅助模型
- 主模式始终是 `AWDP`
- `web` 和 `pwn` 不是独立主模式，而是 AWDP 下可叠加的子模式
- 系统必须模块化，训练库、题型插件、skills、MCP 都要能替换
- 运行环境按 `Windows + WSL2` 设计

系统原则：

- 单主脑：所有最终推理、裁决、补丁方案都由同一个主模型负责
- 模块化：模型、提示词、训练数据、skills、MCP、评测完全解耦
- 插件化：通过子模式叠加能力，而不是切换到另一套孤立系统
- 云训本地跑：训练、蒸馏、LoRA 在外部机器完成，创13负责最终本地部署和离线推理

---

## 2. 主模型策略

### 2.1 首发模型

本地唯一主模型建议固定为：

- `DeepSeek-R1-Distill-Llama-70B`
- 量化规格：`GGUF Q4_K_M`

选择理由：

- 这条线更偏推理，适合 AWDP 的分析、修补、日志归因、利用链判断
- `70B Q4_K_M` 在创13这类大统一内存机器上是可部署的强上限方案
- 本地生态成熟，`LM Studio`、`llama.cpp`、`Ollama` 路线都能承接

### 2.2 本地部署规则

- 本地只部署这一套主模型
- 不常驻第二模型
- 如果未来做微调或蒸馏，目标仍然是服务于这一套主模型，而不是把系统拆成多模型协同

### 2.3 本地运行参数

建议首发参数：

- 上下文默认：`16K`
- 重分析：`32K`
- 不默认开 `64K+`
- 运行端第一阶段：`LM Studio`
- 第二阶段：在需要 API 接入时再补 `Ollama`

## 3. 架构：固定 AWDP 主模式 + 可叠加子模式

### 3.1 模式设计

系统不再维护多个平级主模式，而是固定：

- 主模式：`awdp`

在 `awdp` 主模式下叠加子模式：

- `web`
- `pwn`
- 后续可扩：`reverse`、`crypto`、`misc`

组合方式：

- `awdp`
- `awdp+web`
- `awdp+pwn`
- `awdp+web+pwn`

这意味着：

- 所有题目的默认世界观、输出格式、优先级、补丁目标都来自 `awdp`
- `web` 子模式只补 web 方向的语料、payload、漏洞模式、验证流程
- `pwn` 子模式只补二进制和利用方向的语料、分析流程、exp 模板

### 3.3 核心规则

- `Core/awdp` 负责公共输出规范、服务视角、补丁目标、回归检查
- `Plugins/web` 和 `Plugins/pwn` 只做增量能力
- 插件不能覆盖主模式，只能补充主模式
- 一个题目可以同时激活多个插件

## 4. 训练与微调路线

### 4.1 总原则

训练目标不是“再造一个模型”，而是让这一个 `70B` 主模型在 AWDP 场景下更稳定、更像你的作战方式。

训练路线按四层推进：

1. 数据清洗
2. 提示词与 few-shot 结构化
3. 外部微调 / 蒸馏
4. 本地评测回归

### 4.2 第一阶段：先做数据，不做本地训练

先整理三类数据：

- `Core/AWDP` 公共数据
  - 服务分析
  - 日志归因
  - patch 模板
  - 回归检查单
  - 常见攻防路径
- `web` 插件数据
  - 路由和参数面
  - 常见漏洞模式
  - payload 样例
  - 修补前后对比
- `pwn` 插件数据
  - ELF / PE 基础分析
  - 保护机制
  - 栈堆利用路径
  - exp 骨架
  - 调试失败归因

第一阶段产出：

- `cleaned` 训练数据
- `dev` 调参集
- `eval` 基准集
- `few-shot` 样例库

### 4.3 第二阶段：外部微调

由于本地只保留一个主模型，训练建议这样做：

- 在外部机器上做 `LoRA / QLoRA`
- 训练对象仍然是 `DeepSeek-R1-Distill-Llama-70B` 这条能力路线，或者先在同族/同风格小模型上验证数据价值
- 微调目标只针对：
  - 输出结构
  - 题目拆解顺序
  - patch 决策风格
  - 工具调用偏好
  - AWDP 下的保守性和可回归性

不做的事：

- 不在创13上训练 `70B`
- 不把 `web` 和 `pwn` 训成两套割裂人格
- 不在没有 `eval` 集的情况下盲目开训

### 4.4 第三阶段：蒸馏与数据回灌

这一步不是为了再部署一个小模型，而是为了持续增强主模型体系的数据：

- 用人工高质量题解蒸馏生成训练样本
- 用主模型在标准题上的优质输出生成结构化样本
- 失败 case 单独沉淀到 `hard-negative` 集
- 每次训练只更新外部训练集和提示模板，不破坏主架构

## 5. 数据来源与专项项目

### 5.1 pwn 方向

优先参考：

- `pwncollege/challenges`
  - 地址：<https://github.com/pwncollege/challenges>

优先参考：

  - 地址：<https://github.com/WebGoat/WebGoat>

- `OWASP Juice Shop`
  - 地址：<https://github.com/juice-shop/juice-shop>
  - 用途：现代 Web 漏洞训练环境
- `DVWA`
  - 地址：<https://github.com/digininja/DVWA>
  - 用途：低门槛、可重复的 Web 漏洞训练环境

### 5.3 框架与评测参考

- `Cyber-Zero`
  - 地址：<https://github.com/amazon-science/Cyber-Zero>
  - 用途：安全 agent 工作流参考
- `CyBench`
  - 地址：<https://github.com/andyzorigin/cybench>
  - 用途：网络安全评测思路参考

### 5.4 使用策略

这些项目不直接整仓塞进训练集，而是拆成四层：

- 知识层：漏洞原理、利用条件、修补原则
- 过程层：分析步骤、工具调用、验证过程
- 结果层：payload、exp、patch、日志结论
- 评测层：保留题目和 case，绝不回灌训练

## 6. Skills 设计

skills 必须服务于单主模型，不能默认“交给别的模型做”。

### 6.1 AWDP 核心 skills

- 服务面梳理
- 证据整理
- 攻击面排序
- patch 方案排序
- 回归检查
- 业务影响判断
- 日志归因

### 6.2 web 子插件 skills

- 路由与参数点提取
- 输入面与过滤点识别
- payload 生成
- 回显与错误归因
- 漏洞复现流程
- 最小 patch 建议

### 6.3 pwn 子插件 skills

- 文件信息与保护检测
- 崩溃点分析
- 可控性判断
- 利用路线选择
- exp 骨架生成
- 调试失败归因

规则：

- skills 按 `awdp` 主模式和 `web/pwn` 插件分层
- 同一题目允许加载多个 skills 组
- skills 输出必须结构化，方便后续训练回灌

---

## 7. MCP 设计

MCP 也按主模式和插件分层。

### 7.1 核心 MCP

- `filesystem/search`
  - 读本地知识库、题目目录、源码、日志
- `terminal`
  - 调用 WSL2 基础工具链
- `retrieval`
  - 检索训练语料、题解、模板

### 7.2 web 插件 MCP

- `browser`
  - 本地靶场交互、验证回显
- `terminal`
  - `curl`、目录测试、脚本调用

### 7.3 pwn 插件 MCP

- `radare2`
  - 静态分析
- `terminal`
  - `checksec`、`pwntools`、调试命令

使用原则：

- MCP 默认白名单
- `web` 和 `pwn` 插件只暴露必要工具
- 工具调用日志要保留，方便评测和蒸馏

## 8. 工作流

### 8.1 首发工作流

第一版先做最小可用链路：

1. 安装唯一主模型
2. 建 `Core/awdp`
3. 建 `Plugins/web` 与 `Plugins/pwn`
4. 建第一版评测集
5. 跑基线
6. 再决定是否做外部微调

### 8.2 题目执行方式

#### 纯 AWDP

加载：

- `Core/awdp`

#### Web 题

加载：

- `Core/awdp`
- `Plugins/web`

#### Pwn 题

加载：

- `Core/awdp`
- `Plugins/pwn`

#### 复合题

加载：

- `Core/awdp`
- `Plugins/web`
- `Plugins/pwn`

## 9. 评测标准

评测必须围绕 AWDP 主目标，而不是只看“像不像 CTF 解题 bot”。

评测维度：

- 是否识别正确攻击面
- 是否给出可执行利用链
- 是否给出可落地 patch
- 是否考虑业务回归
- 是否正确调用工具
- 是否在复合题里正确使用 `web/pwn` 插件

数据切分必须固定：

- `train`
- `dev`
- `eval`

`eval` 不允许回灌。

## 10. 分阶段实施

### Phase 0：环境与模型

- 安装 `LM Studio`
- 安装 `WSL2`
- 安装基础工具：`python`、`git`、`docker`、`radare2`
- 下载 `DeepSeek-R1-Distill-Llama-70B GGUF Q4_K_M`

### Phase 1：骨架

- 建 `Core/awdp`
- 建 `Plugins/web`
- 建 `Plugins/pwn`
- 建 `Datasets / Eval / Workflows`

### Phase 2：基线

- 导入第一批 `web/pwn` 数据
- 建 `train/dev/eval`
- 跑基线测试
- 输出失败 case

### Phase 3：外部微调

- 基于基线失败点做 LoRA / QLoRA
- 训练目标仍然服务于单主模型
- 跑回归测试

### Phase 4：持续增强

- 新 case 进入数据管线
- 新插件只做增量，不破坏主模式
- 定期重跑 `eval`

---

## 11. 最终建议

当前直接采用以下方案：

- 本地主模型：`DeepSeek-R1-Distill-Llama-70B Q4_K_M`
- 本地只保留这一套模型
- 主模式：`awdp`
- 子插件：`web`、`pwn`
- 运行环境：`Windows + WSL2`
- 训练路线：先数据和评测，再外部微调，再回归

系统核心不是“找一个 pwn/web 神模”，而是：

- 单强模型
- 固定 AWDP 主模式
- 可叠加 web/pwn 插件
- 可替换数据、skills、MCP
- 有评测闭环
